# 実例ベース検証スキル

## 目的

AIが書いたテストだけに頼らず、**実世界に存在する計算例・判定例**を集めて自作ロジックと照合することで、テストの**網羅性**と**正確性**を高める。

「思いつくテストケース」より「実際に世の中で問われている事例」の方が、ロジックの穴を発見しやすい。

## いつ実行するか

- 新しい計算/判定ロジックを実装した直後
- 既存ロジックを大きく変更した時
- `data-update` スキル実行後（数値や閾値が変わった時）
- リリース前の最終確認

## 実例ベース検証 5ステップ

### Step 1: 一次情報から実例を集める

最優先の照合材料。以下から探す：

- 国税庁タックスアンサー / Q&A / パンフレットの「設例」「計算例」
- 各官公庁の手引き（厚労省、中小企業庁、自治体ホームページ）
- 法令集・通達に含まれる事例
- 公的機関が公開しているシミュレーター等

WebFetch または `src/data/sources/` の抜粋を活用する。

### Step 2: 二次情報から実例を集める

裾野を広げるための照合材料：

- 税理士事務所・行政書士事務所のブログ
- 会計ソフト（freee, MF, 弥生）の解説記事
- 専門書・実務書の例題
- 質問サイト（税金 Q&A 系）の事例

検索クエリ例: `「<対象制度名> 計算例」`、`「<対象制度名> 設例」`、`「<対象制度名> 具体例」`

**最低 5〜10件**は集める。多いほど精度が上がる。

### Step 3: 自作ロジックで再計算

各実例の入力値を自作ツール（`src/core/`）に与え、出力を取得する。
結果を**表形式で比較**：

| # | 出典 | 入力 | 期待値（実例の答え） | 自作ロジックの出力 | 一致 |
|---|---|---|---|---|---|
| 1 | NTA タックスアンサー No.XXXX | ... | 123,456 | 123,456 | ✅ |
| 2 | freee 解説記事 | ... | 50,000 | 49,800 | ❌ |

### Step 4: 不一致の原因を追跡

不一致が見つかったら、**まず自分の実装を疑う**。その上で：

| 状況 | 判断 |
|---|---|
| **一次情報**と不一致 | ほぼ確実に自作ロジックが間違っている → 修正 |
| **二次情報のみ**と不一致 | 二次情報が古い/間違っている可能性。一次情報に当たって判定 |
| 計算前提が違う | 例: 改正前の税率を使った古い記事。正しく分類して検証対象から外す |
| 端数処理の差 | 切り上げ/切り捨て/四捨五入の規定を原典で確認 |

**ネット記事は古い・間違っていることが珍しくない**ので、一次情報を最終判定基準にする。

### Step 5: 検証済み実例をテストケースとして固定

合致した実例（または不一致を解消した実例）は、そのままテストファイルに追加してリグレッション検出に使う：

```ts
// src/core/__tests__/verification.test.ts
describe('実例検証', () => {
  // 実例1: 国税庁タックスアンサー No.XXXX
  // https://www.nta.go.jp/.../xxxx.htm
  test('木造建物（事業用）の耐用年数', () => {
    expect(calculate(...)).toBe(...);
  });

  // 実例2: freee 解説記事
  // https://...
  test('...', () => { ... });
});
```

**コメントに必ず出典URLを書く**こと。後で原典を確認できるようにする。

## 補完テスト手法

実例検証だけでは漏れる観点をカバーする補助手法：

### 境界値テスト
ロジックの分岐点（税率階層の変わり目、控除上限、年数の閾値など）の**前後 ±1** で出力が正しく変わるか確認する。`src/core/` の if 文・switch 文を grep して分岐値を洗い出す。

### 既存ツールとのクロスチェック
- 国税庁「確定申告書等作成コーナー」での計算結果と照合
- 市販会計ソフト（freee, MF, 弥生）の同条件での計算結果と照合
- 公的シミュレーターがあれば優先

### Excel / 手計算での独立実装
同じ計算を別の手段（Excel関数、紙の手計算）で再現し、ロジックと比較。実装上のバグ（演算順序、ビット幅）を発見しやすい。

### 過去年度データの再現
古い税率・閾値を使って当時の正解を再現できるか確認。`dataMetadata.ts` に過去版が残っていれば活用。データ履歴の正確性も同時に検証できる。

### 異常系・極端値
- 0円入力
- 上限を大幅に超過する値
- マイナス値
- 小数点を含む値
- 非常に大きな値（オーバーフロー）

エラーになるべき入力でちゃんとエラーになるか、ガード節が機能しているかを確認。

### 専門家レビュー（任意）
余力があれば税理士・行政書士など実務家にロジックや結果をレビュー依頼する。リリース前の最終確認として有効。

## 判断原則

1. **一次情報 > 二次情報 > AIの直感** を絶対に守る
2. 不一致を見つけたら**まず自分の実装を疑う**
3. ネット記事は古い・間違っていることが珍しくない
4. 「合致した数」より「**網羅した観点**」を重視する
5. 検証済み実例は**テストとして固定化**して資産にする

## 適用可能なツールの種類

このスキルは「公式文書をロジック化するツール」全般に適用できる：

- 税金計算（所得税・法人税・住民税・相続税・贈与税…）
- 社会保険料計算（健康保険・年金・介護・労災…）
- 補助金・助成金の支給額計算
- 経費判定・勘定科目分類
- 許認可・届出の要否判定
- 減価償却・耐用年数判定
- 労基法・労契法の判定
- 建築基準法・都市計画法の判定
- … その他、公式文書を根拠とする判定/計算ツール全般
